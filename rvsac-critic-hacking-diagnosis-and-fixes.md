# RVSAC Critic 과대평가(Value Hacking) 진단과 수정안

이 문서는 학습 로그 분석 중 발견된 "actor가 critic의 근사 오차를 기어올라 `value_mean`을
이론적 상한(0) 너머로 밀어올리는" 현상의 근본 원인을 정리하고, 검토한 수정안들을 비용 대비
효과로 비교한 뒤 구현 우선순위를 제안한다.

## 0. 핵심 요약

- 관측: `value_mean`이 학습 중 이론상 불가능한 양수(최대 +20)로 폭주했다가, 다른 구간에서는
  낮은 `critic_loss`를 유지한 채 오염된 양수 오프셋에 눌러앉는 패턴이 반복 관측됨.
- 근본 원인은 두 가지가 겹친 것: **(A)** `γ`가 1에 가까워 "V 전체에 상수를 더해도 손실이
  거의 안 변하는" 평평한 계곡이 존재하고, **(B)** 롤아웃의 마지막 지점(`h_H`)이 이 계곡의
  교정 혜택(TD(λ))을 구조적으로 못 받는 사각지대라는 점.
- actor의 그래디언트(`∇_h V̄(h_H)`)가 정확히 이 사각지대를 겨냥하기 때문에, critic이
  "무엇을 배우든" actor는 그 오차를 이용해 `h_H`를 밀어붙일 수 있다.
- 검토 결과 **K-step lookahead(롤아웃을 h_H 너머로 몇 스텝 더 굴려 타깃을 보강하는 안)는
  구현 비용 대비 효과가 낮다** — 아래 §3에서 정량적으로 보인다. 대신 **h_H에 별도의
  실측 보상 기반 앵커 손실을 추가하는 안(§2.2)을 1순위로 권장**한다.

## 1. 진단 요약

### 1.1 관측된 현상 (2026-07 학습 로그)

```
step 25900: value_mean=6.33,  return_J= 0.04
step 25950: value_mean=10.39, return_J= 3.97   ← 레짐 전환
...
step 28750: value_mean=13.48, return_J= 6.04   ← 계속 상승
```

동시에 `ce_first`/`ce_last`/`accuracy`는 계속 개선되고 있어, "정책이 실제로 나빠져서"가
아니라 "critic이 이론상 불가능한(양수) 값을 보고하면서 그 크기가 계속 커지는" 현상이었다.

### 1.2 메커니즘: actor의 critic 그래디언트 착취

`rvsac/model.py`의 `terminal_value = self.value_head_target(states[H])`는 파라미터가
얼려있고(`requires_grad_(False)`) `h_H`에만 그래디언트가 흐른다(DDPG의 액터 업데이트와
동일 구조). actor loss `-β·γ^H·V̄(h_H)`를 최소화 = `V̄(h_H)`를 최대화이므로:

```
∇_θ V̄(h_H) = ∇_h V̄(h_H) × ∂h_H/∂θ
```

`∇_h V̄`는 critic이 지금 어느 방향을 후하게 평가하는지(정확하든 근사 오차든 구분 없이)를
가리키고, actor는 그 방향으로 `h_H`를 만드는 파라미터를 옮긴다.

### 1.3 근본 원인 A — `(1-γ)` 평평한 계곡

TD 잔차 `δ_t = r_t + γV(h_{t+1}) - V(h_t)`에서 `V→V+c`(전체에 상수를 더함)로 바꾸면:

```
δ_t' = δ_t - c(1-γ)
```

`γ=0.975`면 `(1-γ)=0.025` — 상수 `c=10`을 더해도 잔차는 겨우 `0.25`만 변한다.
TD(λ)를 쓰면 이 계수가 `(1-γ)/(1-λγ)`로 개선된다(`λ=0.9`일 때 `≈0.204`, 8배 개선).
이 상수-오프셋 방향은 `value_head`의 `fc2.bias` 파라미터 하나로 정확히 구현되는 실제
방향이며, 손실 지형에서 이 방향만 유독 평평한 "골짜기"를 이룬다.

### 1.4 근본 원인 B — `h_H`의 구조적 구멍

`losses.py`의 TD(λ) 재귀:
```python
y[:, end-1] = r_d[:, end-1] + gamma * target_values[:, end-1]      # base case: 항상 순수 1-step
for t in range(end-2, start-1, -1):
    y[:, t] = r_d[:, t] + gamma * ((1-lam)*target_values[:, t] + lam*y[:, t+1])
```

재귀의 시작점(`y[H-1]`)은 뒤에 섞을 `y[H]`가 없어 **`λ`가 뭐든 무조건 순수 TD(0)**이다.
게다가 `model.py`의 `critic_states = states[:-1]`(=`h_0..h_{H-1}`만)라서, **`h_H`는
`value_head`가 직접 회귀 훈련을 받아본 적이 없는 입력**이다(`value_head_target`을 통한
순수 외삽일 뿐). 즉 `h_H`는 (a) 교정력이 제일 약하고 (b) 훈련 이력조차 없는, 시스템에서
가장 무방비한 지점인데, actor의 그래디언트가 정확히 이 지점을 공격한다.

## 2. 검토한 수정안

### 2.1 `V≤0` 페널티 (안전장치, 단독으론 불충분)

```python
value_nonneg_penalty = F.relu(values).square().mean() + F.relu(terminal_online_value).square().mean()
```

`V>0`일 때만 벌점이 걸리는 **한쪽 벽**이다. `V≤0`인 동안은 그래디언트가 정확히 0이라,
"음수 영역 안에서 얼마나 부정확한지"는 전혀 못 잡는다. 증상(양수 노출)은 확실히 막지만
근본 문제(계곡 자체의 평평함)는 안 건드린다. **다른 수정안과 함께, 안전장치로는 유지 가치 있음.**

### 2.2 `h_H` 실측 보상 앵커 손실 — 1순위 권장

```python
# model.py: RVSAC.forward()에 추가
terminal_online_value = self.value_head(states[H].detach())   # 지금까지 한 번도 학습 안 된 입력을 훈련

# losses.py: RVSACLossHead.forward()에 추가
with torch.no_grad():
    eq_anchor_target = r_d[:, -1] / (1 - self.gamma)           # 실측 보상만으로 계산, V 의존성 0
eq_anchor_loss = 0.5 * (terminal_online_value - eq_anchor_target).square().mean()

loss = critic_coef * critic_loss + beta * actor_loss + eq_anchor_coef * eq_anchor_loss
```

이미 진단 지표로만 쓰이던 `eq_gap = |V(h_H) - r_{H-1}/(1-γ)|`을 실제 손실로 승격시킨 것.
`r_{H-1}/(1-γ)`는 **V에 전혀 의존하지 않는** 순수 실측값이라, `V→V+c` 분석을 다시 하면
이 항의 잔차는 `c` 전체가 그대로 드러난다 — **교정 계수 = 1.0** (§3에서 비교).

부수 효과: CE가 아직 안 낮아진(=학습 초반) 상태에선 이 앵커가 상당히 음수인 값을 요구하므로,
"V가 낙관적으로 부풀려는 힘"과 정반대 방향으로 계속 당겨준다 — 관측된 실패 양상(V가 과도하게
낙관적)에 정확히 반대되는 보수적 편향이라 오히려 바람직하다.

### 2.3 K-step lookahead 확장 — 검토했으나 비권장 (§3에서 정량 비교)

롤아웃을 `h_H` 너머로 `K`스텝 더(그래디언트 없이) 굴려서 `y[H-1]`의 타깃을 더 깊은
λ-blend로 만드는 안. 방향은 맞지만, 같은 `λ=0.9`를 계속 쓰는 한 아무리 `K`를 늘려도
교정 계수가 `(1-γ)/(1-λγ)≈0.204`(=롤아웃 중간 지점들과 동일한 수준)에 **점근할 뿐,
2.2의 앵커(계수=1.0)를 절대 못 넘는다.** 게다가 추가 순전파 비용이 실제로 든다.

### 2.4 Twin critic (min 취하기) — 2순위

`value_head`를 독립 초기화된 두 개로 만들고 actor는 `min(V1, V2)`만 본다. 두 critic이
같은 근사 오차 방향을 공유할 확률은 낮으므로, exploit 자체를 어렵게 만드는 **근본 원인 쪽
처방**. 구현 비용은 중간(파라미터·순전파 2배, actor 쪽 로직 수정 필요).

### 2.5 시간척도 분리 강화 — 2순위

`β=0.1`인데도 `td_abs`가 20000+ step 동안 안 줄어든 건 분리가 부족하다는 증거.
`β`를 더 낮추거나(값싼 재튜닝), 혹은 "actor 1스텝당 critic K스텝" 구조로 실제 분리
(중간 비용의 학습 루프 재구성)를 검토.

### 2.6 `γ` 하향 — 최후 수단

`γ=0.975→0.95`면 경계 지점 교정력이 `0.025→0.05`(2배)로 개선되지만, `H` 스텝 전체의
신용 할당 방식 자체가 바뀌는 전역적 변경이라 부작용 범위가 넓다. 1~5번을 넣고도 문제가
남을 때 검토할 손잡이.

## 3. 왜 K-step lookahead보다 §2.2 앵커가 나은가 — 정량 비교

`y[H-1]`의 base case를 `K`스텝 뒤로 미루고 그 구간을 `λ`로 제대로 blend했을 때, 상수
오프셋 `c`에 대한 잔차 교정 계수는 다음 폐형식으로 유도된다(재귀 `s_n=γ(1-λ)c+γλs_{n-1}`,
`s_1=γc`를 풀면):

```
R(K+1) = [(1-γ)/(1-γλ)] × [1 - (γλ)^(K+1)]
```

`γ=0.975, λ=0.9` (`γλ=0.8775`) 기준:

| 방법 | 교정 계수 | 상대 개선(현재 대비) |
|---|---|---|
| 현재 (K=0, 순수 TD(0)) | 0.025 | 1× |
| K=1 (한 스텝 확장) | 0.047 | 1.9× |
| K=4 | 0.098 | 3.9× |
| K=8 | 0.141 | 5.6× |
| K=16 | 0.182 | 7.3× |
| K→∞ (같은 λ 유지) | **0.204** (한계) | 8.2× |
| **§2.2 앵커 (실측 보상만 사용)** | **1.0** | **40×** |

`K`를 아무리 늘려도 `λ=0.9`를 유지하는 한 `0.204`를 절대 못 넘는다(같은 λ가 롤아웃
중간 지점에서 이미 도달한 한계이기 때문). 반면 앵커는 **V 의존성이 아예 없는 실측값**을
쓰므로 `K→∞`보다도 5배 더 강한 교정력을 즉시, 추가 순전파 없이 얻는다.

연산 비용도 비대칭적이다: `K=16` 확장은 `TransitionHead` 순전파를 16번 추가로 태워야
하고(`≈1.8×10¹¹ FLOPs/스텝 × 16 ≈ 2.9×10¹²`, 현재 H=32 풀 BPTT 스텝 비용의 약 17%
추가), 앵커는 `value_head` 순전파 1번(`≈1.3×10⁸ FLOPs`, 무시 가능)만 필요하다.

**결론: K-step lookahead는 방향은 맞지만 "낼 수 있는 최대 효과"가 앵커보다 5배 낮으면서
비용은 훨씬 높다. 우선순위에서 제외하고 §2.2를 채택한다.**

## 4. 권장 구현 순서

1. **§2.2 (`h_H` 앵커 손실)** + **§2.1 (`V≤0` 페널티)** — 동시에, 가장 먼저.
   `config/arch/rvsac.yaml`에 `eq_anchor_coef`, `value_nonneg_coef` 추가(둘 다 `critic_coef`의
   1/10 정도로 작게 시작해 점차 올리며 관찰).
2. 그래도 `td_abs`/`value_mean` 이상이 남으면 **§2.4 (twin critic)**.
3. 여전히 남으면 **§2.5 (시간척도 분리 강화)** — 우선 `β` 재튜닝(값싼 실험)부터.
4. 그래도 남으면 **§2.6 (`γ` 하향)** — 전역 변경이라 마지막 순서.

## 5. 검증 지표

수정 후 다음을 관찰:
- `value_mean`이 다시는 지속적으로 양수를 유지하지 않는지 (일시적 튐은 있어도 빠르게 음수로 복귀하는지)
- `eq_gap`이 학습이 진행되며 실제로 감소 추세를 보이는지 (지금은 정체/악화됐었음)
- `td_abs`가 시간이 갈수록 감소하는지 (지금은 20000+ step 동안 안 줄어듦)
- eval 시 `critic_loss`/`eq_gap`이 train과 비슷한 자릿수로 유지되는지 (지금은 eval에서
  55배 이상 폭증 — 다만 이건 §별개로 데이터셋 일반화 문제와도 얽혀 있어 완전 해소는
  데이터 다양성 문제(1000개 근본 퍼즐 augmentation) 해결과 함께 가야 함, 이 문서 범위 밖)

## 6. 참고: 별도 축인 일반화 문제

이 문서는 critic 동역학만 다룬다. eval 정확도 하락(39%→19.7%)의 상당 부분은 이 문제와
별개로, `data/sudoku-extreme-1k-aug-1000`이 근본적으로 1000개의 서로 다른 스도쿠 문제를
augmentation(숫자 치환/행렬 순열)으로 100만 개 예제로 불려낸 것이라 — augmentation이
논리적 난이도를 안 바꾸는 특성상 일반화 실패를 완전히 막지는 못한다는 별도 원인이 겹쳐
있다. 이건 critic 수정과 무관하게 데이터 다양성(더 큰 subsample-size, 혹은 전체
sudoku-extreme 데이터셋 사용)으로 접근해야 하는 독립적인 문제다.
